# 协议转换生成器输入说明与现有工程配置对照

本文档说明两件事：

- 生成这类 Qt/C++ 协议转换工程时，需要哪些输入
- `newB`、`newC`、`newD` 现有工程里，这些内容分别是如何配置的、写在哪些文件里、各自用来做什么

---

## 1. 输入总览

要完整生成这一类工程，输入建议分为 4 层：

| 层级 | 输入文件/字段 | 作用 |
|---|---|---|
| 协议结构层 | `*.xml` | 定义协议字段、分支、循环、序列 |
| 字段转换层 | `mappings.json -> conversions[].rules` | 定义源字段到目标字段的映射公式 |
| 运行时层 | `mappings.json -> runtime`、`conversions[].runtime` | 定义工程怎么收包、取数、调度、缓存、发送 |
| 业务应答层 | `mappings.json -> runtime.protocol_verifies` | 定义收到消息后如何识别业务状态、是否要回业务应答包 |

这 4 层分别解决：

- XML：协议长什么样
- Rules：字段怎么转
- Runtime：工程怎么跑
- Protocol Verifies：收到后怎么判定、怎么应答

### 1.1 哪些需要配置，哪些可以自动推导

按目前生成器实现，可以这样理解：

| 类别 | 项目 | 说明 |
|---|---|---|
| 必须配置 | XML、字段映射规则 | 没有这两类输入就无法生成协议结构和转换逻辑 |
| 建议配置 | 网络端口、取数次数/等待周期、联合时序矩阵、verify/应答行为 | 这些直接决定工程运行行为是否一致 |
| 可自动推导 | 命名类字段 | 按 `newB` 风格自动生成，不需要显式输入 |

当前可自动推导的命名类字段包括：

- `conversions[].runtime.process_method`
- `conversions[].runtime.sources[].message_name`
- `conversions[].runtime.sources[].display_name`
- `conversions[].runtime.cache_name`
- `conversions[].runtime.cache_num`
- `runtime.protocol_verifies[].constraints[].name`
- `runtime.protocol_verifies[].verify_rules[].name`
- `runtime.protocol_verifies[].response_actions[].on_verify`

也就是说，后续输入重点应该放在“行为”上，而不是“名字”上。

## 2. 协议结构输入

### 2.1 XML 协议文件

**作用**

- 定义协议字段长度
- 定义 XML 分支 `Field`
- 定义 XML 循环 `Group`
- 定义消息序列 `MessCode / PreSeq`
- 生成 `*_def.h`
- 生成 `codec.cpp` 的解码/编码函数骨架

**当前生成器输入位置**

- `--protocol-dir` 指向的目录下全部 `*.xml`

**现有工程中对应位置**

- `newB/`、`newC/`、`newD/` 目录本身不直接保存 XML
- 这些工程里的 `codec.cpp`、`*_def.h` 是由 XML 结构“展开后”的结果
- 也就是说，现有工程是“XML 结构已经被编译进 C++ 代码”

**典型生成结果**

- `*_def.h`
- `readOrigin(...) / writeOrigin(...)`
- `writeSeq_*`
- `check<Type>SeqNum(...)`

---

## 3. 字段转换输入

### 3.1 `mappings.json -> conversions[].rules`

**作用**

- 定义源协议字段如何赋值到目标协议字段
- 支持直接映射、常量赋值、表达式、条件赋值
- 生成 `xxx_to_yyy.cpp`

**当前生成器输入示意**

```json
{
  "conversions": [
    {
      "name": "AtoB",
      "sources": [
        {"alias": "a", "protocol": "A"}
      ],
      "target": {
        "protocol": "B"
      },
      "rules": [
        {
          "target_field": "field_b",
          "formula": "a.field_a",
          "source_fields": ["a.field_a"],
          "rule_type": "direct"
        }
      ]
    }
  ]
}
```

**现有工程中对应位置**

- `newB/s0_1_to_w304.cpp`
- `newB/s106_to_w204.cpp`
- `newB/newC/w304_to_iCD304.cpp`
- `newB/newD/w304_to_iCD304.cpp`

这些文件里写的是“展开后的最终赋值代码”，并不是运行时配置。

## 4. 运行时输入

运行时输入控制的是：

- 网络收发
- 线程轮询
- 取数策略
- 联合转换时序检查
- 目标消息发送或缓存

现有工程里，这一层大部分不是外部配置，而是写死在 `messageconvert.cpp` 中。

### 4.1 `runtime.endpoints`

**作用**

- 配置接收端口、发送端口、IP、网络类型、反馈端口
- 生成 `config.xml`
- 程序启动时由 `main.cpp` 读取，再传给 `messageConvert::start(...)`

**建议输入格式**

```json
{
  "runtime": {
    "endpoints": [
      {
        "ip": "127.0.0.1",
        "port": 3333,
        "type": "udp",
        "recv": 1,
        "feed_back_port": 3333,
        "name": "S106"
      },
      {
        "ip": "127.0.0.1",
        "port": 3336,
        "type": "udp",
        "recv": 0,
        "feed_back_port": 3333,
        "name": "W204"
      }
    ]
  }
}
```

**现有工程中怎么设置**

| 工程 | 配置位置 | 当前值 |
|---|---|---|
| `newB` | `newB/config.xml` | 接收 `3333`，发送 `3336` |
| `newC` | `newB/newC/config.xml` | 接收 `3333`，发送 `3336` |
| `newD` | `newB/newD/config.xml` | 接收 `3333`，发送 `3336` |

**说明**

- 这三套工程这里基本一致
- 这是现有工程里少数真正外部化的配置项

### 4.2 `runtime.loop_sleep_ms`

**作用**

- 控制主转换线程每轮执行后休眠多久
- 影响 CPU 占用和轮询延迟
- 生成到 `messageconvert.cpp` 的 `msgConvertThread()` 中

**建议输入格式**

```json
{
  "runtime": {
    "loop_sleep_ms": 2
  }
}
```

**现有工程中怎么设置**

| 工程 | 配置位置 | 当前值 |
|---|---|---|
| `newB` | `newB/messageconvert.cpp` | `_sleep(2)` |
| `newC` | `newB/newC/messageconvert.cpp` | `_sleep(2)` |
| `newD` | `newB/newD/messageconvert.cpp` | `_sleep(2)` |

**说明**

- 这是工程级轮询周期
- 三套工程当前一致

### 4.3 `runtime.check_data_interval_ms`

**作用**

- 只在联合转换模式下使用
- 控制多久触发一次缓存目标消息有效性检查
- 生成到 `checkDataTimer.start(...)`

**建议输入格式**

```json
{
  "runtime": {
    "check_data_interval_ms": 5000
  }
}
```

**现有工程中怎么设置**

| 工程 | 配置位置 | 当前值 |
|---|---|---|
| `newB` | `newB/messageconvert.cpp` | `checkDataTimer.start(5000)` |
| `newC` | 无 | 不使用 |
| `newD` | 无 | 不使用 |

**说明**

- 只有联合转换工程 `newB` 用到
- `newC/newD` 是单源简单转换，没有这层调度

### 4.4 `conversions[].runtime.process_method`

**作用**

- 指定生成的处理函数名
- 例如 `S106dataPro()`、`W30_4dataPro()`、`JointSumDataPro()`
- 影响 `messageconvert.h/.cpp`

**是否必须输入**

- 否
- 这属于命名类字段，可自动推导

**建议输入格式**

```json
{
  "conversions": [
    {
      "runtime": {
        "process_method": "S106dataPro"
      }
    }
  ]
}
```

**现有工程中怎么设置**

- `newB/messageconvert.cpp`
  - `S0_1dataPro()`
  - `S106dataPro()`
- `newC/messageconvert.cpp`
  - `W30_4dataPro()`
- `newD/messageconvert.cpp`
  - `W30_4dataPro()`

**说明**

- 现有工程是直接把函数名写死在源码里
- 生成器中这项可以省略，省略时按 `newB` 风格自动命名

### 4.5 `conversions[].runtime.sources[].message_name`

**作用**

- 指定运行时缓存/取数时使用的消息名
- 影响 `getData("...")` 的查找键

**是否必须输入**

- 否
- 这属于命名类字段，可自动推导
- 简单模式下默认取源协议名
- 联合模式下默认优先取 `choreography.json -> sources[].cache_key`

**建议输入格式**

```json
{
  "conversions": [
    {
      "runtime": {
        "sources": [
          {
            "alias": "s106",
            "message_name": "S106"
          }
        ]
      }
    }
  ]
}
```

**现有工程中怎么设置**

| 工程 | 配置位置 | 当前值示例 |
|---|---|---|
| `newB` | `newB/messageconvert.cpp` | `getData("S01", ...)`、`getData("S106", ...)` |
| `newC` | `newB/newC/messageconvert.cpp` | `getData("W304", ...)` |
| `newD` | `newB/newD/messageconvert.cpp` | `getData("W304", ...)` |

### 4.6 `conversions[].runtime.sources[].display_name`

**作用**

- 联合转换时，用于时序矩阵比较和编排文件匹配的显示名
- 影响 `msgNameList.append(...)`

**是否必须输入**

- 否
- 这属于命名类字段，可自动推导
- 简单模式下默认取源协议名
- 联合模式下默认优先取 `choreography.json -> sources[].protocol`

**建议输入格式**

```json
{
  "conversions": [
    {
      "runtime": {
        "sources": [
          {
            "alias": "s106",
            "display_name": "S协议.S106"
          }
        ]
      }
    }
  ]
}
```

**现有工程中怎么设置**

- `newB/messageconvert.cpp`
  - `msgNameList.append(u8"S协议.S0_1")`
  - `msgNameList.append(u8"S协议.S106")`

**说明**

- `newC/newD` 不是联合转换，这项基本不用

### 4.7 `conversions[].runtime.sources[].fetches`

**作用**

- 定义每轮转换里，源消息怎么取数
- 每项由两个值组成：
  - `count`: 需要的消息次数/累计次数
  - `cycle_ms`: 相邻消息允许的时间窗口

**建议输入格式**

```json
{
  "conversions": [
    {
      "runtime": {
        "sources": [
          {
            "alias": "w304",
            "fetches": [
              {"count": 1, "cycle_ms": 0},
              {"count": 3, "cycle_ms": 2000},
              {"count": 3, "cycle_ms": 2000}
            ]
          }
        ]
      }
    }
  ]
}
```

**现有工程中怎么设置**

| 工程 | 配置位置 | 当前值 |
|---|---|---|
| `newB` | `newB/messageconvert.cpp` | `counts0_1={1,1,1,1,1,1}`、`cycles0_1={0,0,0,0,0,0}` |
| `newB` | `newB/messageconvert.cpp` | `counts106={3,3}`、`cycles106={100,100}` |
| `newC` | `newB/newC/messageconvert.cpp` | `countw304={1,3,3}`、`cyclew304={0,2000,2000}` |
| `newD` | `newB/newD/messageconvert.cpp` | `countw304={1,3,3}`、`cyclew304={0,2000,2000}` |

**说明**

- 这是三套工程差异最大的运行时输入之一
- 如果不外部化，无法稳定复刻三类项目

### 4.8 建议新增：`conversions[].runtime.fetch_match_mode`

**作用**

- 控制 `getData(...)` 对 `count` 的匹配方式

**建议取值**

| 取值 | 含义 |
|---|---|
| `exact` | 必须 `num == item->num` |
| `at_least` | 达到即可，`num <= item->num` |

**现有工程中怎么设置**

| 工程 | 实际代码 |
|---|---|
| `newB` | `num == item->num` |
| `newC` | `num <= item->num` |
| `newD` | `num <= item->num` |

**说明**

- 现有工程没有独立配置项
- 这是写死在 `getData(...)` 实现里的策略差异

### 4.9 建议新增：`conversions[].runtime.consume_state_mode`

**作用**

- 控制消息消费标记的实现方式

**建议取值**

| 取值 | 含义 |
|---|---|
| `per_name` | 用 `QStringList state` 按名字去重 |
| `binary` | 用 `int state` 做 0/1 标记 |

**现有工程中怎么设置**

| 工程 | 实际代码 |
|---|---|
| `newB` | `QStringList state`，`state.indexOf(name) == -1` |
| `newC` | `int state`，`state == 0` |
| `newD` | `int state`，`state == 0` |

### 4.10 `conversions[].runtime.response_enabled`

**作用**

- 控制收到源消息后，如果 `checkObjMaps(...)` 生成了应答包，是否真正用 UDP 发回去

**建议输入格式**

```json
{
  "conversions": [
    {
      "runtime": {
        "response_enabled": true
      }
    }
  ]
}
```

**现有工程中怎么设置**

- `newB/newC/newD` 都是写死在 `messageconvert.cpp`
- 模式是：
  - 调 `checkObjMaps(ret, sdata, value)`
  - 如果返回值不是 `-1` 且反馈端口有效，就 `soc.writeDatagram(...)`

### 4.11 `conversions[].runtime.send_mode`

**作用**

- 控制目标消息生成后是直接发送，还是先缓存

**建议取值**

| 取值 | 含义 |
|---|---|
| `direct` | 直接调用 `onSendMessage(...)` |
| `cache` | 调 `cacheGeneratedTarget(...)` 或等价缓存逻辑 |

**现有工程中怎么设置**

| 工程 | 实际行为 |
|---|---|
| `newB` | 依据 `code_test::getStatus_41(...)`，有时直接发，有时先缓存 |
| `newC` | 直接发 |
| `newD` | 直接发 |

**说明**

- `newB` 这里还叠加了编排状态判断
- 简单工程 `newC/newD` 相当于固定 `direct`

### 4.12 `conversions[].runtime.cache_name` 与 `cache_num`

**作用**

- 当目标消息需要缓存时，指定缓存名称和编号

**是否必须输入**

- `cache_name`：否，属于命名类字段，可自动推导
- `cache_num`：否，省略时默认 `3`

**现有工程中怎么设置**

| 工程 | 实际代码 |
|---|---|
| `newB` | 在 `pushData(...)` 前构造 `msgDataInfo`，例如 `name="W30_4"`、`num=3` |
| `newC` | 基本不用 |
| `newD` | 基本不用 |

**说明**

- `cache_name` 缺省时，生成器可按目标协议名自动推导
- `cache_num` 缺省时，生成器默认使用 `3`

## 5. 联合编排输入

### 5.1 `choreography.json`

**作用**

- 只用于联合转换
- 定义多个输入协议之间的时序关系
- 定义目标消息接收窗口
- 生成 `to_code_Choreography.h/.cpp`

### 5.2 时序矩阵 `joint_groups[].matrix`

**作用**

- 表示多个输入协议之间的时序关系
- 决定两条源消息能不能组成一次联合转换

**建议输入格式**

```json
{
  "joint_groups": [
    {
      "group_id": "g1",
      "target_id": "W304",
      "sources": ["S01", "S106"],
      "trigger_policy": "all_required",
      "matrix": {
        "unit": "ms",
        "rows": ["S01", "S106"],
        "cols": ["S01", "S106"],
        "values": [
          [0, 200],
          [200, 0]
        ]
      }
    }
  ]
}
```

**现有工程中怎么设置**

- `newB/to_code_Choreography.cpp`
- `messageconvert.cpp` 中通过：
  - `code_test::getSrcTime_41(...)`
  - `code_test::getDstMsg_41(...)`
  - `code_test::getStatus_41(...)`
  来使用这份编排数据

**说明**

- `newC/newD` 没有这一层

### 5.3 `targets[].receive_window_ms`

**作用**

- 定义联合目标消息的有效窗口
- 配合 `checkData(...)` 和 `checkDataTimer` 使用

**现有工程中怎么设置**

- `newB/to_code_Choreography.cpp` 中的目标时间表
- `newB/messageconvert.cpp` 中的 `onCheckDataTimer()`

## 6. verify / 应答规则输入

这一层不是 UDP 协议本身的 ACK 规则，而是运行在 UDP 之上的“业务层应答规则”。

它解决的是：

- 收到这条业务消息后，是否命中某个业务状态
- 命中后要不要回业务应答包
- 回包前需要改哪些字段
- 回哪个 `Seq_*`

### 6.1 `runtime.protocol_verifies`

**作用**

- 按“协议”定义 verify 状态机
- 生成：
  - `checkConstraint_*`
  - `setConstraint_*`
  - `checkVerify_*`
  - `Verify<Type>Seq(...)`
  - `checkObjMaps(...)`

**哪些必须输入**

- 主要需要配置“行为字段”：
  - `constraints[].check`
  - `constraints[].set[]`
  - `verify_rules[].when_seq`
  - `response_actions[].encode_seq`（有回包时）
- 命名类字段可省略，由生成器自动推导：
  - `constraints[].name`
  - `verify_rules[].name`
  - `response_actions[].on_verify`

**建议输入格式**

```json
{
  "runtime": {
    "protocol_verifies": {
      "S106": {
        "constraints": [
          {
            "check": "value.field_a == 1 && value.field_b == 0",
            "set": [
              {"field": "field_x", "value": "1"}
            ]
          }
        ],
        "verify_rules": [
          {
            "when_seq": "Seq_1"
          }
        ],
        "response_actions": [
          {
            "set_constraint": "Constraint1",
            "encode_seq": "Seq_1",
            "return_code": 0
          }
        ],
        "default_return_code": -1
      }
    }
  }
}
```

### 6.2 `constraints`

**作用**

- 表示一个业务约束
- 分两部分：
  - `check`: 判断当前对象是否满足约束
  - `set`: 回应答前如何把对象改成某种状态

**命名是否必须输入**

- 否
- 可省略，省略时自动生成为 `Constraint1`、`Constraint2` ...

**现有工程中怎么设置**

- `newB/codec.cpp`
  - `checkConstraint_1`
  - `setConstraint_1`
  - `checkConstraint2`
  - `setConstraint2`
- `newC/newD/codec.cpp`
  - `checkConstraint_1`
  - `setConstraint_1`
  - `checkConstraint_2`
  - `setConstraint_2`

### 6.3 `verify_rules`

**作用**

- 定义：
  - 哪个 `Seq_*`
  - 再加上哪个约束
  - 会被识别成哪个 verify 标签

**命名是否必须输入**

- 否
- 可省略，省略时自动生成为 `verify1`、`verify2` ...

**现有工程中怎么设置**

- `newB/codec.cpp`
  - `checkVerify_1/2/3(W304&)`
  - `checkVerify_1(W204&)`
  - `checkverify1/2(S106&)`
- `newC/newD/codec.cpp`
  - `checkVerify_1/2/3(W304&)`

### 6.4 `response_actions`

**作用**

- 定义收到某个 verify 标签后要执行的动作：
  - 是否调用 `setConstraint`
  - 编码哪个 `Seq_*`
  - `checkObjMaps(...)` 返回多少

**命名是否必须输入**

- `on_verify` 可省略
- 省略时会按顺序自动绑定到对应的 `verify_rules`
- 但 `encode_seq` 仍然属于行为输入，通常需要显式配置

**现有工程中怎么设置**

- `newB/S106`
  - `checkObjMaps(...)` 中：
    - `verify2 -> setverify1 -> writeSeq_1 -> return 0`
    - `verify1 -> return 0`
- `newB/S0_1`
  - `checkObjMaps(...)` 直接 `return 0`
- `newC/newD/ICD304`
  - `checkObjMaps(...)` 直接 `return 0`

### 6.5 `default_return_code`

**作用**

- `checkObjMaps(...)` 未命中任何应答动作时返回什么

**建议约定**

| 返回值 | 含义 |
|---|---|
| `0` | 收到新数据，继续后续转换 |
| `1` | 收到应答数据，并已构造出业务应答包 |
| `-1` | 错误或忽略，不继续后续流程 |

**现有工程中怎么设置**

- `newB/S106`：未命中返回 `-1`
- `newB/S0_1`：固定返回 `0`
- `newC/newD/ICD304`：固定返回 `0`

---

## 7. 现有工程的配置方式总结

现有三个工程并没有统一的“外部输入配置格式”，而是这样分散设置：

| 内容 | 现有设置位置 |
|---|---|
| 网络端口 | `config.xml` |
| 主循环周期 | `messageconvert.cpp` |
| 联合检查周期 | `messageconvert.cpp` |
| 取数次数/等待周期 | `messageconvert.cpp` |
| `getData` 判定规则 | `messageconvert.cpp` |
| 联合时序矩阵 | `to_code_Choreography.cpp` |
| verify 识别规则 | `codec.cpp` |
| 应答动作 | `codec.cpp` |

也就是说，现有工程的很多“配置”其实是硬编码在 C++ 中的。  
生成器的目标，就是把这些硬编码内容还原成外部输入。

---

## 8. 生成器当前建议的输入分工

为了完整复刻 `newB + newC + newD` 三类语义，建议按下面的分工输入：

| 输入位置 | 负责内容 |
|---|---|
| `*.xml` | 协议结构 |
| `mappings.json -> conversions[].rules` | 字段转换公式 |
| `mappings.json -> runtime.endpoints` | 网络端口 |
| `mappings.json -> runtime.loop_sleep_ms` | 主循环周期 |
| `mappings.json -> runtime.check_data_interval_ms` | 联合检查周期 |
| `mappings.json -> conversions[].runtime.sources[].fetches` | 取数次数/等待周期 |
| `mappings.json -> conversions[].runtime.fetch_match_mode` | `==` 或 `<=` 取数规则，属于行为输入 |
| `mappings.json -> conversions[].runtime.consume_state_mode` | `QStringList` 或 `int` 消费状态，属于行为输入 |
| `choreography.json` | 联合转换时序矩阵与目标窗口 |
| `mappings.json -> runtime.protocol_verifies` | verify/应答规则；命名项可省略，行为项建议输入 |
| 自动推导/默认 | `process_method`、`message_name`、`display_name`、`cache_name`、`cache_num` | 不需要显式输入 |

---

## 9. 结论

如果目标只是“把 XML 转成一个能编译的工程”，输入只要：

- XML
- 字段映射

如果目标是“功能完全复刻 `newB/newC/newD`”，还必须输入：

- 网络端口
- 轮询周期
- 联合检查周期
- 取数次数/等待周期
- `getData` 判定策略
- 联合时序矩阵
- verify/应答状态机的行为字段

下面这些字段不需要再作为必填输入：

- `process_method`
- `sources[].message_name`
- `sources[].display_name`
- `cache_name`
- `cache_num`

否则只能生成“结构类似”的工程，不能稳定生成“行为一致”的工程。
