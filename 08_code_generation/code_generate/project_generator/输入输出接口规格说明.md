# 协议转换项目生成器输入输出接口规格说明

## 1. 文档说明

### 1.1 文档定位

本文档定义协议转换项目生成器面向外部集成方的完整输入输出接口规格，目标是同时明确：

- 调用方需要提交什么输入
- 生成器成功时输出什么产物
- 失败时返回什么错误信息

外部可见输入收敛为 4 类：

- `protocol_xml_dir`
- `conversion_rules_json`
- `conversion_matrix_json`
- `port_config_json`

除端口配置外，生成工程所需的其他协议信息、命名信息和结构信息都应由 `protocol_xml_dir` 自动解析或推导。

### 1.2 适用范围

- 适用于 `project_generator` 的对外输入输出接口设计
- 适用于 simple 单链路转换和 joint 联合转换两类场景
- 不限定调用协议，可用于文件投递、CLI 包装层或后续服务化接口

### 1.3 实施状态

本文档描述的是对外目标输入契约。
当前仓库中的真实实现仍主要使用：

- `mappings.json`
- `choreography.json`
- `runtime.endpoints`

因此，本规格描述的是目标对外接口契约；如果接入当前生成器，需要由适配层映射到现有内部格式。

---

## 2. 接口概述

### 2.1 设计原则

- `protocol_xml_dir` 是协议结构和命名信息的唯一事实来源
- `conversion_rules_json` 只描述“怎么转”，不再承担运行时配置职责
- `conversion_rules_json` 由上游大模型接口直接生成并保存，不要求调用方二次拼装
- `conversion_matrix_json` 只描述“联合场景何时能转”，simple 模式下可以为空
- `port_config_json` 是唯一需要人工显式提供的运行时配置
- 输出接口必须同时给出工程产物和生成结果摘要，便于调用方判断是否可直接交付

### 2.2 接口请求与响应

| 接口部分 | 类型 | 说明 |
|------|------|------|
| 请求 `request` | 对象 | 包含 `protocol_xml_dir`、`conversion_rules_json`、`conversion_matrix_json`、`port_config_json` |
| 成功响应 `response` | 对象 | 包含生成状态、输出目录、文件清单、关键产物摘要、告警信息 |
| 失败响应 `error` | 对象 | 包含错误码、错误阶段、失败原因、定位信息 |

### 2.3 输入总览

| 输入项 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `protocol_xml_dir` | 目录路径或目录对象 | 是 | 协议 XML 所在目录，例如 `K/` |
| `conversion_rules_json` | JSON 文件路径或 JSON 对象 | 是 | 定义源字段到目标字段的映射规则 |
| `conversion_matrix_json` | JSON 文件路径、JSON 对象或 `null` | 条件必填 | joint 模式必填，simple 模式可为空 |
| `port_config_json` | JSON 文件路径或 JSON 对象 | 是 | 定义端点与端口号绑定关系 |

### 2.4 推荐文件组织

```text
input/
├── K/
│   ├── k.xml
│   ├── k1_1.xml
│   ├── k1_2.xml
│   └── k1_7.xml
├── conversion_rules.json
├── conversion_matrix.json
└── port_config.json
```

## 3. 自动提取规则

### 3.1 由 XML 自动获取或推导的信息

下列信息不应再作为外部必填输入，而应由系统从 `protocol_xml_dir` 中解析或结合转换关系推导：

| 信息类别 | 获取方式 |
|------|------|
| 协议类型名 | 从 XML 文件与解析结果确定 |
| 字段结构、分支、循环、序列 | 从 XML 中的 `Item`、`Field`、`Group`、`MessCode` 解析 |
| 编解码结构 | 从 XML 结构自动生成 |
| 默认显示名 | 从协议类型名或 XML 中的协议标识推导 |
| 默认缓存名 | 从目标协议名推导 |
| 默认处理函数名 | 从源/目标协议组合推导 |
| `*_def.h` 与 `codec.*` 所需字段清单 | 从 XML 平铺字段模型生成 |

### 3.2 端口配置的边界

端口配置是唯一人工输入的运行时项，主要负责：

- 接收端口
- 发送端口
- 反馈端口

以下内容不应要求调用方重复填写：

- 协议字段结构
- 显示名称
- 缓存命名
- 编解码序列
- 处理函数命名

## 4. 输入接口定义

## 4.1 `protocol_xml_dir`

### 4.1.1 说明

`protocol_xml_dir` 表示协议 XML 所在目录，是所有协议结构信息的唯一来源。
例如可直接传入 `K/` 目录，目录下包含 `k.xml` 以及多个子消息 XML 文件。

### 4.1.2 约束

| 字段/规则 | 要求 |
|------|------|
| 目录存在性 | 必须存在 |
| 文件编码 | UTF-8 |
| 文件数量 | 目录下至少 1 个 `.xml` 文件 |
| 内容要求 | 必须包含可解析的协议结构定义 |
| 协议结构 | 应包含 `Item`、`Field`、`Group`、`MessCode` 等现有生成器支持的节点 |
| 唯一性 | 同一目录内不允许出现含义冲突的重复协议定义 |

### 4.1.3 作用

- 生成 `*_def.h`
- 生成 `codec.h/.cpp`
- 为转换公式和端口配置提供协议名校验基准
- 为联合矩阵中的消息标识提供引用基础

### 4.1.4 示例

```text
K/
├── k.xml
├── k1_1.xml
├── k1_2.xml
├── k1_6.xml
└── k1_7.xml
```

## 4.2 `conversion_rules_json`

### 4.2.1 说明

`conversion_rules_json` 用于描述字段级转换规则，对应当前实现中的 `mappings.json` 核心转换部分。
该对象只负责“目标字段如何由源字段计算得到”，不再携带运行时端口和杂项命名配置。

该文件的来源约定如下：

- 上游大模型规则生成接口直接生成并保存
- 生成器直接读取该 JSON 文件
- 调用方不需要再把 LLM 输出重组为其他中间对象

### 4.2.2 推荐结构

```json
{
  "version": "1.0",
  "project_name": "demo_simple",
  "conversions": [
    {
      "name": "TempSensorToTempReport",
      "mode": "simple",
      "sources": [
        {
          "alias": "sensor",
          "protocol": "Temp_Sensor"
        }
      ],
      "target": {
        "protocol": "Temp_Report"
      },
      "rules": [
        {
          "target_field": "temperature_c",
          "formula": "sensor.temperature",
          "source_fields": ["sensor.temperature"],
          "rule_type": "direct",
          "when": null,
          "default_value": null,
          "description": "温度直接映射"
        }
      ]
    }
  ]
}
```

### 4.2.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | 是 | 规格版本 |
| `project_name` | string | 是 | 项目标识 |
| `conversions` | array | 是 | 转换链列表 |
| `conversions[].name` | string | 是 | 转换链名称 |
| `conversions[].mode` | enum | 是 | `simple` 或 `joint` |
| `conversions[].sources` | array | 是 | 源协议列表 |
| `conversions[].sources[].alias` | string | 是 | 公式中的变量别名 |
| `conversions[].sources[].protocol` | string | 是 | 必须能在 XML 集合中找到 |
| `conversions[].target.protocol` | string | 是 | 目标协议名，必须能在 XML 集合中找到 |
| `conversions[].rules` | array | 是 | 字段规则列表 |
| `rules[].target_field` | string | 是 | 目标字段名 |
| `rules[].formula` | string | 是 | 受控表达式 |
| `rules[].source_fields` | array | 是 | 公式依赖字段 |
| `rules[].rule_type` | enum | 是 | `const`/`direct`/`expression`/`conditional` |
| `rules[].when` | string/null | 否 | 规则生效条件 |
| `rules[].default_value` | any | 否 | 条件不满足或计算失败时的回退值 |
| `rules[].description` | string | 否 | 规则说明 |

### 4.2.4 约束规则

- `formula` 必须是纯表达式，不能包含语句块或任意 C++ 注入
- `source_fields` 中引用的字段必须都来自 `protocol_xml_dir`
- 同一转换内不允许同一个 `target_field` 被重复赋值
- `conversion_rules_json` 不应再包含旧版 `runtime.endpoints`、`loop_sleep_ms` 等外部运行时字段

## 4.3 `conversion_matrix_json`

### 4.3.1 说明

`conversion_matrix_json` 用于描述联合转换时的时序矩阵和目标触发关系。

- simple 模式：可为空或不提供内容
- joint 模式：必填

### 4.3.2 推荐结构

```json
{
  "version": "1.0",
  "mode": "joint",
  "joint_groups": [
    {
      "group_id": "joint_sum",
      "target_protocol": "Target_Sum",
      "sources": ["Source_A", "Source_B"],
      "trigger_policy": "all_required",
      "receive_window_ms": 200,
      "matrix": {
        "unit": "ms",
        "rows": ["Source_A", "Source_B"],
        "cols": ["Source_A", "Source_B"],
        "values": [
          [0, 200],
          [200, 0]
        ]
      }
    }
  ]
}
```

### 4.3.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | 是 | 规格版本 |
| `mode` | string | 是 | joint 模式固定为 `joint` |
| `joint_groups` | array | 是 | 联合触发组列表 |
| `joint_groups[].group_id` | string | 是 | 触发组唯一标识 |
| `joint_groups[].target_protocol` | string | 是 | 目标协议名，必须存在于 XML 集合 |
| `joint_groups[].sources` | array | 是 | 参与联合的源协议名列表 |
| `joint_groups[].trigger_policy` | enum | 是 | `all_required`/`any_required`/`min_count` |
| `joint_groups[].receive_window_ms` | integer | 是 | 联合结果有效窗口 |
| `matrix.unit` | string | 是 | 当前固定 `ms` |
| `matrix.rows` | array | 是 | 行协议列表 |
| `matrix.cols` | array | 是 | 列协议列表 |
| `matrix.values` | array | 是 | 二维阈值矩阵 |

### 4.3.4 约束规则

- `rows` 和 `cols` 必须与 `sources` 保持一致
- 对角线必须为 `0`
- 非空矩阵值必须大于等于 `0`
- `null` 表示该两条消息之间不做时序比较
- simple 模式下若提供该对象，其内容必须为空或显式标注未启用

## 4.4 `port_config_json`

### 4.4.1 说明

`port_config_json` 是唯一需要人工显式提供的运行时输入，用于定义端口、消息类型以及转发过滤约束。
接口8会基于该对象自动派生内部 `runtime.endpoints`，不再接受旧版 `ports/endpoints` 结构。

### 4.4.2 推荐结构

```json
{
  "messageType": "k",
  "recvIp": "192.168.23.16",
  "recvPort": 4100,
  "sendIp": "192.168.23.20",
  "sendPort": 5100,
  "messageRuleDetailList": [
    {
      "messageName": "K1_7",
      "delayRequirement": 2,
      "filterConfig": {
        "crcCheck": {
          "enabled": true,
          "bindElement": "威胁形式"
        },
        "loopConfig": {
          "type": "NONE"
        },
        "aggregation": {
          "mode": "SINGLE",
          "count": null,
          "timeMs": null
        },
        "aggregationType": {
          "type": "TIME",
          "bindElement": null
        }
      }
    }
  ]
}
```

### 4.4.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `messageType` | string | 是 | 消息类别标识，用于生成运行时接收端名称 |
| `recvIp` | string | 是 | 接收端 IP |
| `recvPort` | integer | 是 | 接收端端口 |
| `sendIp` | string | 是 | 发送端 IP |
| `sendPort` | integer | 是 | 发送端端口 |
| `messageRuleDetailList` | array | 是 | 消息级转发与过滤配置 |
| `messageRuleDetailList[].messageName` | string | 是 | 规则作用的消息名，通常对应目标协议名 |
| `messageRuleDetailList[].delayRequirement` | integer | 否 | 延迟要求，默认 `0` |
| `messageRuleDetailList[].filterConfig` | object | 否 | 过滤配置对象；缺省时按默认值补全 |
| `messageRuleDetailList[].filterConfig.crcCheck.enabled` | boolean | 否 | 是否启用 CRC 计算 |
| `messageRuleDetailList[].filterConfig.crcCheck.bindElement` | string | 条件必填 | `crcCheck.enabled=true` 时必须提供；绑定需要计算 CRC 的字段名 |
| `messageRuleDetailList[].filterConfig.loopConfig.type` | enum | 否 | `NONE`/`VALUE`/`CONDITION` |
| `messageRuleDetailList[].filterConfig.aggregation.mode` | enum | 否 | `SINGLE`/`COUNT`/`TIME` |
| `messageRuleDetailList[].filterConfig.aggregation.count` | integer | 否 | `mode=COUNT` 时使用 |
| `messageRuleDetailList[].filterConfig.aggregation.timeMs` | integer | 否 | `mode=TIME` 时使用 |
| `messageRuleDetailList[].filterConfig.aggregationType.type` | enum | 否 | `TIME`/`ORDER`/`DISTINCT` |
| `messageRuleDetailList[].filterConfig.aggregationType.bindElement` | string | 否 | 排序绑定字段 |

### 4.4.4 默认规则

未显式提供的过滤子项由适配层补齐：

- `loopConfig.type`: `NONE`
- `aggregation.mode`: `SINGLE`
- `aggregation.count`: `null`
- `aggregation.timeMs`: `null`
- `aggregationType.type`: `TIME`
- `aggregationType.bindElement`: `null`
- `crcCheck.enabled`: `false`
- `crcCheck.bindElement`: `null`

内部派生的端点默认值：

- `type`: `udp`
- `feedback_port`: 使用 `recvPort`
- `recv`：接收端固定 `1`，发送端固定 `0`

### 4.4.5 约束规则

- 端口号范围必须为 `1-65535`
- `messageRuleDetailList` 至少包含 1 条规则
- `crcCheck.enabled=true` 时必须提供 `bindElement`
- 旧版 `ports/endpoints` 结构不再接受

## 5. 输出接口定义

## 5.1 成功响应 `response`

### 5.1.1 说明

当生成成功时，输出接口应至少返回一份结果摘要，用来描述生成是否完成、产物放在哪里、有哪些关键文件以及是否存在非阻断告警。

### 5.1.2 推荐结构

```json
{
  "status": "success",
  "project_name": "demo_joint",
  "mode": "joint",
  "output": {
    "project_dir": "output/project",
    "files": [
      "main.cpp",
      "messageconvert.cpp",
      "messageconvert.h",
      "codec.cpp",
      "codec.h",
      "config.xml",
      "protocol_manifest.json"
    ],
    "conversion_units": [
      "source_a_source_b_to_target_sum.cpp",
      "source_a_source_b_to_target_sum.h"
    ]
  },
  "warnings": [],
  "summary": {
    "protocol_count": 3,
    "conversion_count": 1,
    "joint_group_count": 1
  }
}
```

### 5.1.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `status` | enum | 是 | 固定为 `success` |
| `project_name` | string | 是 | 生成项目名 |
| `mode` | enum | 是 | `simple` 或 `joint` |
| `output.project_dir` | string | 是 | 生成工程目录 |
| `output.files` | array | 是 | 产物文件清单 |
| `output.conversion_units` | array | 否 | 具体转换单元文件 |
| `warnings` | array | 否 | 非阻断告警 |
| `summary.protocol_count` | integer | 是 | 参与生成的协议数量 |
| `summary.conversion_count` | integer | 是 | 转换链数量 |
| `summary.joint_group_count` | integer | 否 | 联合组数量，simple 可为 `0` |

## 5.2 输出工程产物

### 5.2.1 必备产物

| 产物 | 说明 |
|------|------|
| `main.cpp` | 生成工程入口 |
| `messageconvert.h/.cpp` | 运行时收发、取数、调度逻辑 |
| `codec.h/.cpp` | 协议编解码逻辑 |
| `config.xml` | 网络端口配置产物 |
| `protocol_manifest.json` | 协议解析与生成摘要 |
| `*_def.h` | 协议字段定义头文件 |
| `xxx_to_yyy.h/.cpp` | 字段转换实现文件 |

### 5.2.2 joint 模式附加产物

| 产物 | 说明 |
|------|------|
| `to_code_Choreography.h/.cpp` | 联合编排和时序矩阵实现 |

### 5.2.3 推荐输出目录结构

```text
output/project/
├── main.cpp
├── config.xml
├── peach.pro
├── messageconvert.cpp
├── messageconvert.h
├── codec.cpp
├── codec.h
├── protocol_manifest.json
├── *_def.h
├── xxx_to_yyy.cpp
├── xxx_to_yyy.h
└── to_code_Choreography.cpp   # 仅 joint 模式
```

## 5.3 失败响应 `error`

### 5.3.1 推荐结构

```json
{
  "status": "error",
  "stage": "validation",
  "code": "INVALID_MATRIX_REFERENCE",
  "message": "conversion_matrix_json 中引用的协议名未在 protocol_xml_dir 中找到",
  "details": {
    "field": "joint_groups[0].sources[1]",
    "value": "Source_B"
  }
}
```

### 5.3.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `status` | enum | 是 | 固定为 `error` |
| `stage` | enum | 是 | `parse`/`validate`/`render`/`write` |
| `code` | string | 是 | 错误码 |
| `message` | string | 是 | 可读错误信息 |
| `details.field` | string | 否 | 具体失败字段路径 |
| `details.value` | any | 否 | 导致失败的值 |

## 5.4 典型错误码

| 错误码 | 含义 |
|------|------|
| `INVALID_XML` | XML 文件无法解析 |
| `UNKNOWN_PROTOCOL` | 输入中引用了不存在的协议 |
| `DUPLICATE_TARGET_FIELD` | 同一目标字段被重复赋值 |
| `INVALID_FORMULA` | 转换公式不符合受控表达式约束 |
| `INVALID_MATRIX_REFERENCE` | 转换矩阵引用非法协议名 |
| `INVALID_PORT` | 端口号不合法或冲突 |
| `RENDER_FAILED` | 模板渲染失败 |
| `WRITE_FAILED` | 输出文件写入失败 |

## 6. 接口校验规则

### 6.1 通用校验

- 4 类输入对象必须属于同一个项目上下文
- `conversion_rules_json`、`conversion_matrix_json`、`port_config_json` 中引用的协议名必须都能在 `protocol_xml_dir` 中找到
- 所有文件必须为 UTF-8 编码

### 6.2 simple 模式校验

- 必须提供 `protocol_xml_dir`
- 必须提供 `conversion_rules_json`
- 必须提供 `port_config_json`
- `conversion_matrix_json` 可以为 `null`

### 6.3 joint 模式校验

- 必须提供 `protocol_xml_dir`
- 必须提供 `conversion_rules_json`
- 必须提供 `conversion_matrix_json`
- 必须提供 `port_config_json`
- `conversion_matrix_json` 中所有矩阵引用都必须与多源转换链保持一致

## 7. 示例

## 7.1 simple 场景请求

```text
input/
├── protocols/
│   ├── temp_sensor.xml
│   └── temp_report.xml
├── conversion_rules.json
├── conversion_matrix.json   # null 或空对象
└── port_config.json
```

说明：

- `protocol_xml_dir` 提供 `Temp_Sensor` 和 `Temp_Report` 的协议结构
- `conversion_rules_json` 提供温度字段映射公式
- `conversion_matrix_json` 不参与 simple 场景
- `port_config_json` 仅指定收发端口

### 7.1.1 simple 场景成功输出

```json
{
```

## 8. 与当前实现的映射关系

| 目标输入接口 | 当前实现对应位置 |
|------|------|
| `protocol_xml_dir` | `--protocol-dir` 下的 XML 文件集合 |
| `conversion_rules_json` | `mappings.json` 中的 `conversions[].rules` 及关联转换信息 |
| `conversion_matrix_json` | `choreography.json` 中的 `joint_groups[].matrix` 及联合目标窗口 |
| `port_config_json` | `mappings.json -> runtime.endpoints`，最终生成 `config.xml` |

| 目标输出接口 | 当前实现对应位置 |
|------|------|
| `response.output.project_dir` | `build --output` 指定的输出目录 |
| `response.output.files` | 输出目录中的生成文件清单 |
| `response.summary` | `protocol_manifest.json` 与构建过程统计信息 |
| `error` | 当前实现尚未统一结构化，建议由适配层统一封装 |

### 8.1 当前实现兼容说明

- 当前实现仍允许在 `mappings.json` 中携带更多 `runtime.*` 字段
- 本规格要求对外调用方不再显式暴露这些杂项字段
- 当前实现已具备真实输出工程目录与文件产物，但尚未统一输出结构化 `response/error` 对象
- 如需接入当前生成器，可在适配层中把本规格转换为现有 `mappings.json/choreography.json` 结构，并把构建结果封装成本文定义的输出对象

---

## 9. 一句话结论

> 对外接口显式定义 4 类输入和 2 类输出：输入只接收 `protocol_xml_dir`、`conversion_rules_json`、`conversion_matrix_json`、`port_config_json`，输出则统一返回生成产物摘要或结构化错误信息；除端口配置外，其余生成所需信息全部以 `protocol_xml_dir` 为源自动提取或推导。
